Registry / API Architecture для RPG-Core

Данный документ фиксирует единый архитектурный принцип регистрации и использования игровых систем в RPG-Core. Его задача — определить общий стандарт для всех будущих систем ядра и аддонов, чтобы дальнейшая разработка шла по одному шаблону и не приводила к разрозненным реализациям.

RPG-Core развивается как модульный RPG-движок. Это означает, что ядро должно предоставлять общие интерфейсы, registries, managers и runtime-состояние, а весь контент и большая часть игровых механик должны добавляться через независимые модули. Для соблюдения этого подхода в проекте вводится единый registry/API принцип.

Основной принцип архитектуры RPG-Core следующий:

Registry хранит описание типа. Runtime хранит состояние конкретного экземпляра.

Это ключевое правило всей архитектуры.

Если объект представляет собой описание игрового типа, он должен регистрироваться через registry. Если объект представляет собой конкретное состояние в мире, у игрока, у NPC, у гильдии или у квеста, он не должен быть registry-объектом и должен храниться как runtime data через соответствующий manager, storage или profile.

В качестве примеров registry-объектов уже выступают классы и перки. В текущем ядре это реализовано через RpgClass, RpgPerk, RpgClassRegistries и RpgPerkRegistries. Эти объекты описывают типы игрового контента, а не состояние конкретного игрока.  

В качестве примера runtime-состояния уже выступает RpgProfile. Он хранит уровень, опыт, выбранный класс, выбранные перки и баланс кошелька конкретного игрока. Это не registry-объект, а состояние конкретного экземпляра игрока.  

На основе этого принципа в RPG-Core фиксируются следующие архитектурные уровни.

Первый уровень — API-interfaces.

API-интерфейсы описывают, как выглядит определённый тип игрового контента. Они должны быть лёгкими, стабильными и не зависеть от конкретной реализации аддонов. Через такие интерфейсы ядро и аддоны договариваются о формате данных.

Примеры уже существующих API-интерфейсов:
RpgClass
RpgPerk

Примеры будущих API-интерфейсов:
RpgMarkerType
RpgProfession
RpgQuestType
RpgNpcRole
RpgFactionType

API-интерфейс должен описывать только данные и поведение, относящиеся к самому типу. Он не должен хранить состояние конкретного игрока, конкретного NPC или конкретного объекта мира.

Второй уровень — Registries.

Каждый тип игрового контента, который должен быть расширяемым, должен иметь собственный registry. Registry отвечает за регистрацию, поиск и получение описаний игровых типов по ResourceLocation.

Уже существующие примеры:
RpgClassRegistries
RpgPerkRegistries

Будущие примеры:
RpgMarkerRegistries
RpgProfessionRegistries
RpgQuestRegistries
RpgNpcRoleRegistries

Во всех registries в качестве идентификатора должен использоваться только ResourceLocation. Это обязательный стандарт ядра. Не допускается ввод альтернативных строковых идентификаторов, числовых id или специальных внутренних форматов там, где объект относится к типу игрового контента.

Третий уровень — Core Managers / Services.

Managers и services отвечают за runtime-логику. Они работают не с описанием типа как таковым, а с состоянием игры и игровыми операциями.

Уже существующие примеры:
RpgLevelingService

Будущие примеры:
RpgEconomyService
RpgStorageManager
RpgMarkerManager
RpgQuestManager

Manager не должен подменять registry. Его задача — не хранить описание типов, а управлять состоянием, операциями, синхронизацией и логикой использования этих типов в игре.

Четвёртый уровень — Runtime Data.

Runtime data — это конкретное состояние. Это данные, которые принадлежат игроку, NPC, гильдии, фракции, квесту, контейнеру, карте или другой конкретной сущности.

Уже существующий пример:
RpgProfile

Будущие примеры:
RpgStorage
RpgMarkerInstance
RpgQuestProgress
RpgFactionReputationState
RpgGuildData

Runtime data не регистрируется через registry, потому что это не описание типа, а конкретный экземпляр данных.

На уровне проектирования всех новых систем вводится обязательное разделение между типом и экземпляром.

Например, в системе маркеров:
RpgMarkerType — это registry-тип
RpgMarkerInstance — это конкретный маркер в мире

В системе профессий:
RpgProfession — это registry-тип
RpgProfessionProgress — это состояние игрока по данной профессии

В системе NPC:
RpgNpcRole — это registry-тип роли
NpcData или аналогичная runtime-структура — это конкретный NPC в мире

В системе storage:
тип владельца и логика доступа могут описываться через единый стандарт ядра, но конкретное хранилище конкретной гильдии или NPC должно быть runtime-объектом, а не registry-типом.

Для всех будущих систем RPG-Core фиксируются следующие обязательные правила.

Первое правило.
Все игровые типы, которые должны расширяться ядром или аддонами, регистрируются только через registry.

Второе правило.
Все runtime-состояния хранятся только как data-объекты и управляются через manager/service/storage layer.

Третье правило.
Во всех registries используется только ResourceLocation как идентификатор типа.

Четвёртое правило.
API-интерфейсы не должны зависеть от конкретной реализации аддонов и не должны содержать runtime-состояние.

Пятое правило.
Managers не должны становиться “вторыми registries”. Они не описывают типы, а управляют их использованием.

Шестое правило.
Если система содержит и тип, и конкретные экземпляры, она обязательно должна быть разделена на registry-layer и runtime-layer.

Седьмое правило.
Все новые системы ядра и аддонов должны проектироваться в соответствии с этим разделением. Исключения не допускаются, если только нет отдельного архитектурного обоснования.

Данный принцип напрямую влияет на дальнейшие этапы разработки.

Система Storage Core будет реализована как runtime-система хранения, управляемая через manager. Она не должна проектироваться как registry контента. Registry может понадобиться только для стандартизации типов владельцев, режимов доступа или форматов, если это действительно потребуется, но не для хранения конкретных складов.

Система Marker System будет реализована как комбинация registry-типов маркеров и runtime-экземпляров маркеров.

Система NPC в будущем должна разделяться на registry-типы ролей, поведения и шаблонов с одной стороны, и конкретных NPC в мире с другой стороны.

Система Quests должна разделяться на описание квеста и прогресс конкретного игрока по данному квесту.

Система Professions должна разделяться на описание профессии и прогресс игрока.

Система Guilds и Factions должна разделяться на описание типов и правил, если такие будут нужны, и на конкретные данные гильдий, репутации, казны и хранилищ.

Таким образом, registry/API architecture становится единым фундаментом для всех будущих систем проекта.

RPG-Core должен развиваться не как набор отдельных механик, а как единый расширяемый движок, где все типы контента имеют общий формат регистрации, а все игровые состояния имеют общий формат runtime-управления.

Обновлённая дорожная карта ядра

Ниже фиксируется актуальная дорожная карта именно по ядру, с учётом текущего состояния v7.2.2.

Этап 1–7 считаются выполненными в ранее согласованном объёме. На текущий момент в ядре уже реализованы:
профиль игрока, XP-система, уровни, классы, перки, базовый GUI, HUD, сетевой обмен и базовая экономика кошелька с командами и переводами. Это подтверждается кодом и структурой v7.2.2.

Дальнейшая работа по ядру фиксируется в следующем порядке.

Этап 8. Архитектурное завершение Core.

Подэтап 8.2.0 — Registry/API principle.
Зафиксировать единый архитектурный стандарт ядра. Этот подэтап настоящим документом считается выполненным.

Подэтап 8.2.1 — Storage Core.
Реализовать универсальную систему runtime-хранилищ для игроков, гильдий, фракций, NPC и будущих систем. Система должна строиться как runtime-layer с manager-архитектурой и не должна нарушать зафиксированный registry/API принцип.

Подэтап 8.2.2 — Marker System Core.
Реализовать базовую систему маркеров и координатных точек как основу для карты, квестов, NPC и сюжетных объектов.

Подэтап 8.2.3 — GUI Framework Foundations.
Выделить общие правила построения окон, секций, панелей, кнопок, карточек, scroll-областей и reusable GUI-паттернов, которые будут использоваться в будущих системах ядра и аддонов.

Подэтап 8.2.4 — Core hooks / event expansion.
Расширить базовые события и точки расширения ядра для дальнейших аддонов: NPC, Quests, Professions, Guilds, Map, Cinematics.

Этап 9. Final Core Visual Pass.

После того как core systems будут закрыты, выполняется единый финальный проход по внешнему виду.

В этот этап входит:
финальный дизайн RpgCharacterScreen
финальный дизайн RpgClassScreen
финальный дизайн RpgPerksScreen
финальный визуал HUD
будущий визуал mana bar
единый стиль кнопок, рамок, подложек, декоративных элементов и иконок

Именно на этом этапе делается итоговый art pass ядра. До завершения core systems полный дизайн не финализируется, чтобы не переделывать интерфейсы после каждой архитектурной правки.

Этап 10. Переход к аддонам.

Только после завершения ядра и финального visual pass начинается реализация крупных модулей:
Classes & Perks Addon
NPC Addon
Cinematics Addon
Professions Addon
Factions & Guilds Addon
Map & Navigation Addon

Таким образом, дальнейший порядок работы фиксируется окончательно:

сначала — архитектурный стандарт
затем — Storage Core
затем — Marker System
затем — GUI/HUD final pass
затем — крупные аддоны
