RPG Core Engine — актуальная дорожная карта и фиксация состояния проекта

1. Назначение проекта

RPG Core Engine — это инфраструктурное ядро для модульной RPG-системы в Minecraft.

Это не контентный мод, а именно движок / core framework, на базе которого строятся отдельные аддоны и игровые системы.

Ядро должно содержать только:
 • системные слои
 • API
 • runtime-механику
 • хранение данных
 • экономику
 • storage
 • права доступа
 • базовые GUI/HUD foundations
 • сетевую синхронизацию
 • универсальные механики, которые нужны многим аддонам

Ядро не должно содержать:
 • конкретных NPC как контент
 • сюжет
 • конкретные города
 • конкретные квесты как контент
 • готовые игровые фракции как контентные сущности
 • готовые игровые сцены
 • наполненные торговые точки
 • прикладное наполнение мира

Итоговый принцип:
 • Core = infrastructure
 • Addons = gameplay systems and content over infrastructure

2. Главный архитектурный принцип проекта

Во всём проекте запрещено смешивать:
 • описание системы
 • runtime-состояние

Базовое правило:
 • registry = definition
 • runtime = instance state

Это правило обязательно соблюдать для:
 • classes
 • perks
 • storage
 • access rules
 • dialogues
 • quests
 • services
 • markers
 • cutscenes
 • NPC definitions

Это один из основных архитектурных столпов проекта и его нельзя нарушать в дальнейшей разработке.

⸻

3. Неподвижные архитектурные ограничения

В проекте всегда должны сохраняться следующие правила:

3.1. Ядро не становится аддоном

Ядро должно оставаться инфраструктурным и не должно самовольно превращаться в content-mod.

3.2. Нельзя смешивать core и addon logic

Если логика относится к конкретному аддону или игровому наполнению, она не должна “просачиваться” в core.

3.3. HUD / GUI — обязательная часть ядра

Визуальная часть не считается второстепенной.
HUD, GUI foundations, service GUI style и UI/UX polish — обязательные части ядра, а не “когда-нибудь потом”.

3.4. Anti-dupe и server-authoritative design должны учитываться заранее

Особенно для:
 • economy
 • storage
 • service GUI
 • NPC services
 • shops
 • bankers
 • guild services

Нельзя проектировать будущие сервисы так, чтобы появлялись дыры под:
 • лаги
 • повторные запросы
 • двойное исполнение операций
 • дюп предметов/денег

3.5. GitHub — основной источник истины

Реальный код в GitHub считается основным состоянием проекта.
Если нужно что-то подтвердить по реализации, сначала смотрится реальный код, а не предположение “по памяти”.

3.6. Все игровые операции должны быть server-authoritative

Любые действия, влияющие на:
 • экономику
 • предметы
 • storage
 • сервисы (покупки, банки, NPC)

должны выполняться на сервере.

Клиент не должен иметь возможности:
 • инициировать выполнение без серверной проверки
 • повторно исполнять действие
 • обходить access/economy ограничения

В будущем обязательно:
 • request ids
 • one-shot execution
 • atomic application of results


4. Утверждённые аддоны проекта

На текущий момент согласованы только следующие аддоны:
 1. Classes & Progression Addon
 2. NPC Addon
 3. Guild Addon
 4. Map / Minimap Addon
 5. Cutscene Addon

Нельзя самовольно добавлять новые “утверждённые аддоны”, если они отдельно не зафиксированы пользователем.

5. Что важно помнить по каждому утверждённому аддону

5.1. Classes & Progression Addon

Это аддон поверх class/perk/xp/profile foundation ядра.

Он должен использовать:
 • Class API
 • Perk API
 • XP API
 • profile/progression hooks

Он отвечает за:
 • расширенную систему развития
 • более глубокую прокачку
 • дополнительный progression layer поверх базового ядра

5.2. NPC Addon

Это большой системный аддон уровня не хуже CustomNPC, но глубже, удобнее и сильнее интегрированный с ядром.

В него входят:
 • NPC system
 • dialogues
 • trading
 • bankers
 • service NPC
 • guild NPC
 • faction-related NPC logic
 • routes, patrols, behaviour
 • интеграция со storage
 • интеграция с economy
 • интеграция с access/permission
 • базовый и расширенный функционал уровня CustomNPC+

Критически важно:
в NPC Addon обязательно входит базовая quest system.

Минимум для quest-части:
 • выдача квестов через NPC
 • отслеживание состояния
 • интеграция с диалогами
 • интеграция с наградами
 • интеграция с маркерами
 • деление квестов по типам:
 • важные
 • второстепенные
 • дополнительные / побочные

Если позже появятся повторяемые квесты — это уже расширение.
Но базовая quest system обязана входить именно в NPC Addon.

5.3. Guild Addon

Это отдельный аддон.

Он отвечает за:
 • создание гильдий
• членство
 • роли
 • ранги
 • права
 • доступы к guild storage
 • интеграцию с NPC Addon
 • интеграцию с economy
 • интеграцию с access/permission system

5.4. Map / Minimap Addon

Это отдельный аддон.

Он отвечает за:
 • мини-карту
 • карту мира
 • отображение сюжетных NPC
 • отображение зон квестов
 • отображение точек интереса
 • навигационные элементы для игрока

Это обязательное направление и оно уже отдельно зафиксировано.

5.5. Cutscene Addon

Это отдельный аддон.

Он отвечает за:
 • систему кат-сцен
 • последовательности сцен
 • запуск сцен по условиям/событиям
 • интеграцию с NPC, диалогами, квестами, действиями

Критически важно:
Cutscene Addon обязан иметь editor.

Редактор должен позволять:
 • создавать сцену
 • задавать этапы
 • задавать условия
 • задавать действия
 • определять участников
 • настраивать последовательность выполнения

10. Правила работы с кодом

При любой дальнейшей разработке соблюдать следующий порядок:
 1. Сначала проверять GitHub-репозиторий JambyED/rpg_core_engine
 2. Не гадать по памяти, если файл можно прочитать из репозитория
 3. Если меняется существующий файл — при риске поломки давать полный код под замену
 4. Если нужен новый файл — явно указывать путь и имя
 5. Не давать опасные фрагменты вместо полной замены большого файла
 6. Не перепрыгивать через foundation-steps
 7. Не ломать существующую архитектуру
 8. Не смешивать core и addon logic
 9. Не забывать, что ядро строится под будущие аддоны, но само контентом не становится

10.1. Нельзя создавать параллельные системы

Если в ядре уже существует subsystem (access, storage, economy и т.д.),
запрещено создавать альтернативную реализацию рядом.

Все расширения должны идти через:
 • существующие API
 • существующие сервисы
 • существующие модели

GitHub — это не резервная копия, а основной источник истины по проекту и контроль состояния разработки.

11.1. Среда работы

Используется:

Корень проекта:

11.2. После каждого завершённого значимого этапа обязательно:
 1. Зафиксировать изменения через git
 2. Дать полный набор команд для PowerShell
 3. Указать версию
 4. Описать, что фиксируется
 5. Не переходить к следующему этапу без фиксации предыдущего

cd "C:\Users\jacki\OneDrive\Desktop\MinecraftDev\rpg_core\forge-mdk"

git add .
git commit -m "vX.Y.Z - <краткое описание завершённого этапа>"
git push
git tag vX.Y.Z

Если нужно сначала подтянуть изменения:

git status

11.4. Обязательные правила git-процесса
 • версии должны идти последовательно
 • описание коммита должно отражать реально завершённый этап
 • нельзя пропускать фиксацию
 • нельзя продолжать разработку без фиксации предыдущего состояния
 • каждый этап разработки должен завершаться git-фиксацией

Разработка без фиксации в GitHub считается нарушением процесса и не допускается.
